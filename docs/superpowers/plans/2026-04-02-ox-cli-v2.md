# ox-cli: Ratatui Agentic Coding CLI

## Context

We're building the native CLI target for the ox agent framework. The browser shell (ox-web) already works — this is the terminal counterpart. The key architectural insight driving the design: there is no distinction between "built-in" and "external" tools. All tools are of a kind, go through the same `Tool` trait API, and are instances of a single closure-backed struct.

This means the first change isn't in the CLI crate at all — it's adding `FnTool` to ox-kernel as the canonical way to create a tool from a closure. Then refactoring ox-gate's `CompletionTool` to use it (proving the universality). Then building the CLI on top.

## Changes

### 1. Add `FnTool` to ox-kernel

**Modify:** `crates/ox-kernel/src/lib.rs` (after `Tool` trait, line 233)

Add `FnTool` — a closure-backed `Tool` impl:

```rust
pub struct FnTool {
    name: String,
    description: String,
    schema: serde_json::Value,
    run: Box<dyn Fn(serde_json::Value) -> Result<String, String> + Send + Sync>,
}

impl FnTool {
    pub fn new(
        name: impl Into<String>,
        description: impl Into<String>,
        schema: serde_json::Value,
        run: impl Fn(serde_json::Value) -> Result<String, String> + Send + Sync + 'static,
    ) -> Self {
        Self { name: name.into(), description: description.into(), schema, run: Box::new(run) }
    }
}

impl Tool for FnTool {
    fn name(&self) -> &str { &self.name }
    fn description(&self) -> &str { &self.description }
    fn parameters_schema(&self) -> serde_json::Value { self.schema.clone() }
    fn execute(&self, input: serde_json::Value) -> Result<String, String> { (self.run)(input) }
}
```

Update the doc example on the `Tool` trait to show `FnTool::new` as the primary way to create tools.

Add a unit test.

### 4. Transport (`crates/ox-cli/src/transport.rs`)

Three functions:
- `build_request(provider, api_key, request) -> Result<(url, headers, body_string), String>`
- `parse_response(provider, body) -> Vec<StreamEvent>`
- `make_send_fn(provider, api_key) -> impl Fn(&CompletionRequest) -> Result<Vec<StreamEvent>, String> + Send + Sync`

Uses `reqwest::blocking::Client`, dispatches by provider string, delegates SSE parsing to `ox_gate::codec::{anthropic, openai}`.

Unit tests: verify request construction and response parsing for both providers (no real HTTP).

### 5. Standard tools (`crates/ox-cli/src/tools.rs`)

Single file, ~150 lines. Contains:

- `resolve_path(workspace, relative) -> Result<PathBuf, String>` — path normalization + sandbox check
- `read_file(workspace) -> FnTool` — reads a file, schema: `{path}`
- `write_file(workspace) -> FnTool` — writes a file (creates parent dirs), schema: `{path, content}`
- `edit_file(workspace) -> FnTool` — string replacement, errors if `old_string` not found or not unique, schema: `{path, old_string, new_string}`
- `shell(workspace) -> FnTool` — `sh -c` in workspace dir, returns stdout+stderr+exit code, schema: `{command}`
- `standard_tools(workspace) -> Vec<Box<dyn Tool>>` — returns all four boxed

Unit tests with `tempfile::tempdir()` for each tool.

### 6. App orchestration (`crates/ox-cli/src/app.rs`)

- `AppEvent` enum: `Agent(AgentEvent)`, `Done(Result<String, String>)`
- `ChatMessage` enum: `User(String)`, `AssistantChunk(String)`, `ToolCall { name }`, `ToolResult { name, output }`, `Error(String)`
- `App` struct with TUI state + `mpsc` channel endpoints
- `App::new(provider, model, max_tokens, api_key, workspace)` spawns background agent thread

The agent thread composes directly (no `Agent` from ox-core — it's not `Send`):
- `Kernel::new(model)`
- `Namespace` with `SystemProvider`, `HistoryProvider`, `ToolsProvider`, `ModelProvider`
- `GateStore` mounted at `"gate"`, API key written, `create_completion_tools(send)` registered — the agent can delegate to other models
- Blocks on `prompt_rx.recv()`, runs `kernel.run_turn()` per prompt, sends `AppEvent`s over `event_tx`

### 7. TUI (`crates/ox-cli/src/tui.rs`)

- `run(app, terminal)` — event loop with 50ms crossterm poll + mpsc drain
- `draw(frame, app)` — layout: title (1) | messages (fill) | input (3) | status (1)
- Key bindings: Enter submit, Esc/Ctrl+C quit, Up/Down scroll, standard text editing

### 8. Main wiring (`crates/ox-cli/src/main.rs`)

Wire Clap → resolve API key (arg or env var) → resolve workspace → `App::new` → `ratatui::init()` → `tui::run()` → `ratatui::restore()`.
